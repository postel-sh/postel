> **Pre-alpha.** Receiver-side `v0.1.0` is the only thing currently shipping code. Most pages on this site document the spec; not all of it is implemented yet. See [implementation status](/docs/status) once it lands.

Postel is a **library, not a service.** It runs inside the host application against the host application's database — Postgres or SQLite — and ships in multiple languages over time. Every port conforms to the same wire format, DB schema, and capability behaviors, verified end-to-end by the [`@postel/compliance`](https://github.com/postel-sh/postel/tree/main/compliance) test suite.

## Positioning

> Svix is for when webhooks are your product. Postel is for when webhooks are a feature of your product.

## What's documented here

This site is generated from the canonical specs in the [Postel monorepo](https://github.com/postel-sh/postel):

- **Capability behaviors** — the [`openspec/specs/*`](https://github.com/postel-sh/postel/tree/main/openspec/specs) tree, one page per capability.
- **Wire format** — [AsyncAPI 3.0](https://github.com/postel-sh/postel/blob/main/specs/wire-format/asyncapi.yaml).
- **DB schema** — [canonical SQL DDL](https://github.com/postel-sh/postel/blob/main/specs/db-schema/0001_init.sql).
- **Architectural decisions** — the [`decisions/`](https://github.com/postel-sh/postel/tree/main/decisions) ADRs.
- **Compliance corpus** — the YAML vectors that gate every port.

Each page carries an implementation-status badge. Today, only `@postel/edge` (receiver) is `Available`. Everything else is `Spec stable · not implemented`.
