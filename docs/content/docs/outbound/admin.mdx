description: A framework-agnostic HTTP control plane over the outbound sender тАФ endpoints, replay, key rotation тАФ with default-deny auth and tenant scoping.

`@postel/admin` turns your configured `Postel` instance into a privileged HTTP control plane. `adminRouter(postel, { authorize })` returns a Web `(Request) => Promise<Response>` that maps REST routes to `postel.outbound.*`.

```ts title="lib/admin.ts"
import { adminRouter } from "@postel/admin";
import { postel } from "./postel";

export const admin = adminRouter(postel, {
  // boolean | { allow, status?, tenantId? } | Promise<...>
  authorize: (req) => checkAdminToken(req.headers.get("authorization")),
});
```

## Mounting

The web adapters mount the router for you with `hwa.admin.bindAdminRoutes(prefix, opts)` тАФ `authorize` flows straight through:

```ts
// Hono / Express / Fastify тАФ same shape
HonoWebAdapter(postel, app).admin.bindAdminRoutes("/admin", {
  authorize: (req) => checkAdminToken(req.headers.get("authorization")),
});
```

Or mount the `adminRouter` Fetch handler yourself:

```ts
app.all("/admin/*", (c) => admin(c.req.raw));   // Hono тАФ Fetch-native

import { fetchToExpress } from "@postel/express";
app.use("/admin", fetchToExpress(admin));        // Express

import { fetchToFastify } from "@postel/fastify";
app.all("/admin/*", fetchToFastify(admin));       // Fastify
```

## Authorization тАФ default-deny

`authorize` is effectively required: with none configured, **every request is 403** (and logs once). It receives the raw `Request`, so read whatever you authenticate with тАФ bearer token, mTLS header, cookie. Return `true`/`false`, or `{ allow, status?, tenantId? }` to choose `401` vs `403` and bind the caller to a tenant.

When `authorize` returns a `tenantId`, every route is scoped to it: lists and creates are constrained, and by-id routes return **404** (not 403) for another tenant's resources тАФ so the API never leaks which ids exist in other tenants. Prefer transport auth in front? Run it and pass `authorize: () => true`.

## Routes

| Method ┬╖ path | Does |
|---|---|
| `GET /endpoints` ┬╖ `POST /endpoints` | list ┬╖ create |
| `GET/PATCH/DELETE /endpoints/:id` | get ┬╖ update ┬╖ delete |
| `POST /endpoints/:id/disable` | disable (pause) |
| `POST /endpoints/:id/rotate-secret` | rotate signing secret |
| `POST /replay` ┬╖ `POST /reconcile` | replay ┬╖ reconcile |
| `POST /tenants/:id/rate-limit` ┬╖ `DELETE /tenants/:id` | tenant rate limit ┬╖ delete |
| `POST /keys/symmetric` ┬╖ `POST /keys/asymmetric` | generate signing keys |

Failures map to status by `PostelError.code` тАФ `ENDPOINT_NOT_FOUND` тЖТ 404, `ENDPOINT_VALIDATION` тЖТ 422, `ENDPOINT_DISABLED` тЖТ 409, `MIGRATION_REQUIRED` тЖТ 503 тАФ with a `{ errorCode, error }` JSON body. Code-only endpoint options (`filter`, `transform`, callable `headers`) aren't configurable over HTTP; set them in your `Postel` config.

> Listing events/attempts and viewing raw payloads aren't part of the control-plane router yet тАФ they land when the outbound surface gains a message/attempt query API.
