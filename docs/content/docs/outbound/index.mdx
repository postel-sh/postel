> **Status: planned — v0.2.0.** The outbound API is *designed* (capability specs, TypeScript types, ADRs) but **not implemented at runtime**. Calling any `postel.outbound.*` method today throws `NotImplementedError`. This section documents the *target* shape so you can evaluate whether v0.2 will solve your problem when it lands.

The outbound half of Postel delivers Standard Webhooks to your customers' HTTP endpoints. It's the half you reach for when *your* product emits webhooks — order.created, deployment.finished, message.posted — and someone else's app is the receiver.

If you're verifying webhooks that other services send to you — that's the [inbound half](/docs/inbound), and it ships today.

## What v0.2 will deliver

| Feature | Why it matters |
|---|---|
| **Transactional outbox** | `send()` writes an outbox row inside *your* business transaction. The webhook is queued only if your business write commits — no orphaned events, no "we charged the card but never told the customer." |
| **Retries with backoff** | Exponential or linear backoff, configurable per endpoint, with circuit breakers and auto-disable on persistent failure. |
| **Replay as a first-class verb** | `postel.outbound.replay({ messageId })` or `replay({ endpointId, since })`. No queue surgery, no manual SQL. |
| **Fanout** | One `send()` → N endpoints, filtered by event type, channel, or arbitrary predicate. |
| **Endpoint lifecycle** | `endpoints.create/update/delete/list/get/disable`, with `rotateSecret({ keepPreviousFor })` for overlap-window rotation. |
| **Key management** | Symmetric (`generateSymmetric`) and asymmetric (`generateAsymmetric`) key generation. JWKS publication via the inbound [`jwksHandler`](/docs/inbound/key-rotation#publishing-a-jwks-producer-side). |
| **Multi-tenancy** | Per-tenant rate limits, isolated noisy-neighbor scheduling. |
| **Worker strategies** | In-process workers by default; pluggable for BullMQ, pg-boss, or external workers. |
| **KMS integration** | Sign with AWS KMS, GCP KMS, Vault — secrets never leave the boundary. |
| **Observability** | OpenTelemetry instrumentation across the send pipeline. |

## The mental model

The same `Postel({...})` factory accepts an `outbound` config alongside `inbound`. The two are independent — configure one, both, or neither.

```ts
import { Postel, HmacV1, ExponentialBackoff, InProcess } from "@postel/core";

const postel = Postel({
  outbound: {
    storage: /* your DB connection / ORM, plugged via a storage adapter */,
    signing: HmacV1(),
    retryPolicy: ExponentialBackoff({ maxAttempts: 8 }),
    workers: InProcess({ concurrency: 4 }),
  },
});

// Inside a transaction:
await db.tx(async (tx) => {
  await db.orders.insert({ id: "ord_123", /* ... */ }, { tx });
  await postel.outbound.send(
    { type: "order.created", data: { id: "ord_123" } },
    { tx },                                  // joins the same transaction
  );
});
```

The key insight: `send()` is just an outbox INSERT under the hood. It commits or rolls back atomically with your business write. A separate worker picks the outbox row up and does the HTTP delivery — but you, the application author, never block on the network inside your transaction.

## When does it ship?

Honest answer: no firm date. The v0.2 milestone is gated on:

- All sender-side capability specs (sender, retry-policy, replay-reconciliation, endpoint-management, key-management, filtering-transformation, observability) finalized — in progress.
- The `@postel/compliance` test suite covering sender behaviors (in progress).
- The TypeScript runtime implementing the suite end-to-end.
- The full storage-adapter matrix (`drizzle`, `kysely`, `prisma`, plus standalone clients).

Until then: the API surface is stable enough that we ship the *types*. You can write code today against the types — TypeScript will catch shape mistakes — but the runtime throws `NotImplementedError`. This is intentional: it lets adopters design their integration before the runtime lands.
