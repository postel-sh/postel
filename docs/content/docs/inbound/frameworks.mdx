---
title: Framework adapters
description: Adapter status per framework. Hono ships today; others planned. The framework-agnostic factory works everywhere.
---

`@postel/core` is framework-agnostic — anywhere you can hand it raw bytes and headers, the `Postel({...})` factory works. Framework adapters add two niceties on top:

- They handle the raw-bytes plumbing for you (no `arrayBuffer()` / `req.rawBody` dance).
- They expose the configured `Postel` instance on the framework's request context, so you don't have to import the singleton in every route file.

Until your framework's adapter ships, the factory snippet from [Verify a signed request](/docs/inbound/verify) works against the framework's native request type directly.

## Status

| Framework | Package | Ships? | Notes |
|---|---|---|---|
| Hono | `@postel/hono` | ✅ Today | `verifyWebhook()` / `honoAdapter(postel).verify()` gate middleware + `withWebhook()` wrapper, built on `@postel/http`. (`honoVerify`/`postelHono` kept as deprecated aliases.) |
| Express | `@postel/express` | ⏳ Soon | Package exists as a stub; full middleware planned. |
| Fastify | `@postel/fastify` | ⏳ Soon | Stub today. |
| Next.js Route Handlers | `@postel/nextjs` | ⏳ Soon | Stub today. Native `Request` works fine without an adapter; the package is for type ergonomics. |
| Bun.serve | `@postel/bun` | ⏳ Soon | Stub today. |
| Deno.serve | (none planned) | n/a | Use the factory directly with `req.arrayBuffer()`. |
| SvelteKit, Astro, Nitro | (none planned) | n/a | Use the factory directly. |

> "Stub today" means the package is registered (`@postel/express`, etc.) but contains only a name export — no real runtime. Don't depend on these stubs in production; they exist to reserve the name in the workspace.

## Hono — the real adapter

```ts title="lib/postel.ts"
import { Postel, Secret } from "@postel/core";

export const postel = Postel({
  inbound: {
    vendor: { verify: Secret(process.env.WEBHOOK_SECRET!) },
  },
});
```

```ts title="app.ts"
import { Hono } from "hono";
import { verifyWebhook, honoAdapter, POSTEL_CONTEXT_KEY } from "@postel/hono";
import { postel } from "./lib/postel";

const app = new Hono();

// Gate the route with the source you configured. Verification runs before your
// handler; SIGNATURE_INVALID / TIMESTAMP_TOO_OLD / MALFORMED_HEADER map to the
// right HTTP status automatically, and your handler simply never runs on failure.
app.post("/webhooks/vendor", verifyWebhook(postel.inbound.vendor), (c) => {
  const { event } = c.get(POSTEL_CONTEXT_KEY); // verified · typed · raw bytes intact
  return c.json({ ok: true, type: event.type });
});

// Or bind by source key through the adapter object — the key is type-checked
// against the sources you configured:
const wh = honoAdapter(postel);
app.post("/webhooks/github", wh.verify("github"), (c) => c.text("ok"));
```

The gate reads the raw request bytes (no JSON re-serialization), runs the verifier(s) you configured on that source, and on failure short-circuits with the mapped status — a non-`PostelError` (e.g. a programming bug) bubbles as 5xx rather than being miscast as a 4xx. For a single self-contained route, `withWebhook(postel.inbound.vendor, (c) => …)` folds the gate and your handler into one.

The error→status policy and the byte-preservation plumbing live once in [`@postel/http`](/docs/reference/packages); every framework adapter binds to it. The older raw-secret helpers `honoVerify` / `postelHono` remain as deprecated aliases.

See [`@postel/hono` source](https://github.com/postel-sh/postel/tree/main/typescript/packages/frameworks/hono) for the exposed API.

## Other frameworks (until adapters ship)

The factory call works directly against any runtime that exposes raw bytes:

```ts title="Next.js Route Handler"
export async function POST(req: Request) {
  const body = new Uint8Array(await req.arrayBuffer());
  const headers = Object.fromEntries(req.headers);
  const { event } = await postel.inbound.vendor.verify(body, headers);
  return new Response("ok");
}
```

```ts title="Express (with express.raw)"
import express from "express";
const app = express();

app.post(
  "/webhooks/vendor",
  express.raw({ type: "application/json" }),
  async (req, res) => {
    const body = new Uint8Array(req.body);
    const headers = req.headers as Record<string, string>;
    const { event } = await postel.inbound.vendor.verify(body, headers);
    res.status(200).send("ok");
  },
);
```

```ts title="Fastify (with content-type-parser)"
fastify.addContentTypeParser(
  "application/json",
  { parseAs: "buffer" },
  (_req, body, done) => done(null, body),
);

fastify.post("/webhooks/vendor", async (req, reply) => {
  const body = new Uint8Array(req.body as Buffer);
  const headers = req.headers as Record<string, string>;
  const { event } = await postel.inbound.vendor.verify(body, headers);
  return reply.send("ok");
});
```

```ts title="Bun.serve"
Bun.serve({
  async fetch(req) {
    if (req.method !== "POST") return new Response(null, { status: 405 });
    const body = new Uint8Array(await req.arrayBuffer());
    const headers = Object.fromEntries(req.headers);
    const { event } = await postel.inbound.vendor.verify(body, headers);
    return new Response("ok");
  },
});
```

The full adapters will package each of these as one-liner middleware (and add framework-specific runtime detection for `RawBytesMismatchDetected`). Until then, the snippets above are what we run in production.
